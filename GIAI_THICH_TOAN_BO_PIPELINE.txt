Tài liệu này mô tả cách hệ thống đang vận hành theo code hiện tại của repo.
Phạm vi mô tả bám theo các file chính:
- README.md
- config/sources.yml
- app/pipelines/common/*
- app/pipelines/news|price|weather|policy|traffic/*
- app/models/entities.py
- app/services/*
- app/api/*

==================================================
1. TỔNG QUAN LUỒNG END-TO-END
==================================================

Luồng tổng thể của hệ thống hiện tại là:

Nguồn dữ liệu -> Load cấu hình source -> Fetch payload -> Lưu raw storage -> Parse ra record chuẩn hóa -> Lưu clean data vào database -> Service/API đọc dữ liệu -> AI chat trả lời -> UI Streamlit hiển thị.

Tham chiếu:
- README.md
- app/pipelines/common/base.py
- scripts/run_pipeline.py

Repo hiện có 5 pipeline ingestion:
- news
- price
- weather
- policy
- traffic

Ngoài ingestion, repo còn có:
- FastAPI backend
- Streamlit UI
- AI chat qua POST /chat/query
- Scheduler local
- Cleanup dữ liệu cũ theo retention
- Retrieval experimental cho news và policy

==================================================
2. CẤU HÌNH NGUỒN DỮ LIỆU
==================================================

Tất cả source được định nghĩa trong:
- config/sources.yml

Mỗi source hiện mô tả các trường chính:
- name: tên source duy nhất
- pipeline: source thuộc pipeline nào
- source_type: rss, json, xml, html
- url: URL để fetch live
- parser: parser nào sẽ xử lý payload
- category_default: category mặc định
- active: có bật source hay không
- fetch_interval_minutes: tần suất scheduler
- timeout_seconds
- retry_count
- demo_fixture: file fixture nếu có
- demo_only_source: source chỉ dùng khi chạy --demo-only
- max_items: số item ingest mỗi lần
- max_age_days: cửa sổ dữ liệu theo ngày cho source nào cần retention ở ingest
- các trường extra khác như:
  - query_params
  - headers
  - form_data
  - location_name
  - type_map
  - product_map
  - field_default
  - location_keywords

Cấu hình này được load bởi:
- app/pipelines/common/source_loader.py

source_loader sẽ đọc YAML và chuyển từng item thành SourceDefinition.
Các trường chuẩn được tách riêng.
Các trường còn lại được giữ trong source.extra để parser hoặc service dùng tiếp.

==================================================
3. DATABASE, RAW STORAGE VÀ CÁCH CHỌN DATABASE RUNTIME
==================================================

Phần database runtime nằm ở:
- app/db/session.py

Hành vi hiện tại:
- Nếu DATABASE_URL là SQLite thì dùng luôn SQLite.
- Nếu DATABASE_URL là PostgreSQL thì hệ thống thử probe kết nối.
- Nếu probe PostgreSQL thất bại và môi trường không phải production, hệ thống fallback sang SQLite.

SQLite fallback mặc định dùng để local/demo nhanh.
PostgreSQL dùng khi cần runtime chính.

Schema dữ liệu chính nằm ở:
- app/models/entities.py

Các bảng chính:
- sources
- crawl_jobs
- raw_documents
- articles
- article_clusters
- price_snapshots
- weather_snapshots
- policy_documents
- traffic_events
- document_embeddings

Raw storage file nằm trên filesystem, được quản lý bởi:
- app/pipelines/common/raw_storage.py

Khi fetch thành công, payload raw được ghi theo cấu trúc:
- data/raw/<pipeline>/<source>/<YYYY>/<MM>/<DD>/<timestamp_hash>.<ext>

Đồng thời metadata raw cũng được ghi vào bảng:
- raw_documents

==================================================
4. LUỒNG CHẠY CHUNG CỦA MỘT PIPELINE
==================================================

Luồng chạy chung nằm ở:
- app/pipelines/common/base.py
- scripts/run_pipeline.py

Bước 1: chọn pipeline để chạy
- scripts/run_pipeline.py nhận:
  - --pipeline all|news|price|weather|policy|traffic
  - --demo-only
  - --source <source_name>

Bước 2: đảm bảo schema local nếu đang dùng SQLite
- scripts/run_pipeline.py gọi ensure_sqlite_schema()

Bước 3: khởi tạo pipeline tương ứng
- NewsPipeline
- PricePipeline
- WeatherPipeline
- PolicyPipeline
- TrafficPipeline

Bước 4: BasePipeline.run() load danh sách source
- load_sources_for_pipeline(self.pipeline_name)
- nếu có --source thì lọc theo source name

Bước 5: skip source nếu cần
- Nếu source có demo_only_source=true mà không chạy --demo-only thì skip
- Nếu đang chạy --demo-only nhưng source không có demo_fixture thì skip

Bước 6: sync source vào bảng sources
- SourceRepository.sync_from_definition(...)

Bước 7: tạo crawl job mới
- JobRepository.start_job(...)

Bước 8: fetch dữ liệu
- fetch_source() trong app/pipelines/common/fetcher.py

Hành vi fetch_source():
- nếu không phải demo_only và source có URL thì thử fetch live trước
- có timeout, retry, logging
- nếu live lỗi và có demo_fixture, đồng thời settings cho phép fallback thì dùng fixture
- nếu chạy --demo-only thì đọc fixture luôn

Fetcher hiện dùng:
- httpx.request(...)
- có thể gửi GET hoặc POST
- có thể dùng query params hoặc form_data

Bước 9: lưu raw payload
- RawStorage.save_text(...) ghi file raw
- RawDocumentRepository.create(...) ghi metadata vào raw_documents

Bước 10: parse payload
- mỗi pipeline có hàm parse riêng

Bước 11: store record sạch vào database
- mỗi pipeline có hàm store riêng

Bước 12: cập nhật crawl job
- JobRepository.finish_job(...)
- lưu:
  - status
  - total_fetched
  - total_success
  - total_failed
  - error_message
  - metadata_json như used_demo, total_skipped

Kết quả cuối cùng của mỗi source được trả về dưới dạng PipelineRunSummary.

5. PIPELINE NEWS
==================================================

File chính:
- app/pipelines/news/pipeline.py
- app/pipelines/news/parser.py

Nguồn hiện tại:
- VnExpress RSS: tin mới, thời sự, kinh doanh
- Dân Trí RSS: trang chủ, thế giới, kinh doanh
- Thanh Niên RSS: trang chủ, thời sự, kinh tế
- Tuổi Trẻ RSS, có enrich detail HTML
- Demo fixture RSS cho local/demo

Luồng parse:
- Parser dùng feedparser để đọc RSS
- Hàm chính:
  - parse_news_feed(...)

Với RSS cơ bản:
- lấy title
- lấy summary/description
- lấy content nếu có
- lấy author
- lấy published_at
- lấy canonical_url
- tạo article_hash
- tạo cluster_key

Với Tuổi Trẻ:
- parse RSS trước
- với mỗi bài, fetch thêm HTML detail nếu không phải fixture demo
- trích:
  - summary
  - content_clean
  - author
  - published_at

Retention ở ingest:
- max_age_days=30 cho toàn bộ source news hiện tại
- parser sẽ bỏ các bài cũ hơn 30 ngày nếu feed còn trả về
- logic này dùng helper chung:
  - is_datetime_within_age_window(...)
  trong app/pipelines/common/processing.py

Store của news:
- NewsPipeline.store(...)

Các bước store:
- lấy recent articles 48 giờ gần nhất để dedup nhanh
- bỏ bài nếu canonical_url đã tồn tại
- bỏ bài nếu article_hash đã trùng trong recent set
- so sánh similarity_score(title) để đánh dấu near_duplicate
- gắn cluster bằng article_clusters
- ghi vào bảng articles

Điểm đặc biệt:
- Mỗi article sau khi lưu còn được chia chunk bằng split_into_chunks(...)
- Tạo sparse vector bằng build_chunk_vectors(...)
- Ghi vào document_embeddings
- Đây là index experimental cho retrieval topic_summary

  - parse XML

- PricePipeline.store(...)
- không dedup theo item
- mỗi lần crawl sẽ ghi thêm snapshot mới vào price_snapshots

Ý nghĩa:
- price là dữ liệu time-series snapshot

11. RETRIEVAL EXPERIMENTAL
==================================================

Các file chính:
- app/services/retrieval_service.py
- app/services/retrieval_index_service.py
- app/repositories/embedding_repository.py

Hiện tại retrieval hoạt động theo cách local-first:
- không dùng pgvector
- dùng sparse vector JSON lưu trong bảng document_embeddings

Index được build ở đâu:
- news pipeline khi lưu article
- policy pipeline khi lưu policy document
- hoặc backfill bằng script:
  - scripts/build_retrieval_index.py

RetrievalService:
- search_news_documents(query)
- search_policy_documents(query)

Cơ chế:
- build sparse vector cho query
- lấy toàn bộ chunk embedding cùng doc_type
- tính cosine score đơn giản
- lọc theo min_score
- lấy hit tốt nhất cho mỗi document

Hiện retrieval chỉ nối vào:
- topic_summary của news
- policy_lookup ngữ nghĩa khi keyword search chưa đủ tốt

Nếu retrieval tắt hoặc lỗi:
- hệ thống fallback về flow keyword/search cũ

15. SCHEDULER LOCAL
==================================================

Các file chính:
- app/services/scheduler_service.py
- scripts/run_scheduler.py

Scheduler hiện là local scheduler đơn giản, chưa phải distributed scheduler.

Cách hoạt động:
- load toàn bộ source active từ config/sources.yml

- interval của job lấy từ fetch_interval_minutes

Trạng thái job được lưu ở:
- data/processed/scheduler_status.json

Thông tin theo dõi gồm:
- last_started_at
- last_finished_at
- last_status
- last_total_fetched
- last_total_success
- last_total_failed
- last_error_message
- run_count
- health_state
- failure_streak
- last_duration_seconds
- next_run_at

Scheduler có thể:
- chạy một vòng bằng --run-once
- show trạng thái bằng --show-status
- chạy kèm cleanup bằng --cleanup-after-run

- chạy bằng --demo-only

==================================================
19. KẾT LUẬN NGẮN
==================================================

Hệ thống hiện tại không chỉ là 5 parser riêng lẻ.
Nó đã là một flow đầy đủ gồm:
- cấu hình source
- fetch live hoặc fixture
- raw storage
- clean storage trong DB
- retention và cleanup
- service/API
- AI agent fallback + OpenAI
- UI Streamlit
- scheduler local
- retrieval experimental cho news và policy

Về mặt vận hành, cách hiểu ngắn gọn nhất là:
